CPU使用率过高的问题

上次某公司面试官问了我一个实际生产问题：CPU 使用率达到 100% 怎么办，如何排查和解决。

$CPU使用率=1-空闲时间/总CPU时间$。其中空闲时间通常记为 idle，而总 CPU 时间由用户态时间、低优先级用户态时间、内核态时间、等待 I/O 时间、硬中断时间、软中断时间和 steal 时间等部分组成。

排查时不要只看一个总百分比，应该先判断 CPU 时间主要花在什么地方：us 高通常说明用户态计算密集；sy 高说明内核态开销大，可能是系统调用、锁竞争或内核网络栈消耗；wa 高说明 CPU 在等 I/O；hi、si 高则要关注硬中断或软中断，例如网卡包量过大；st 高多见于虚拟化环境，说明宿主机正在抢占这台虚拟机的 CPU。

一个比较稳的排查顺序如下：

  1. 用 top 或 htop 看整体 CPU、负载、运行队列长度，并定位占用 CPU 最高的进程。
  2. 用 pidstat -u -p <pid> 1 观察目标进程的用户态和内核态 CPU 比例。如果是多线程程序，再用 top -H -p <pid> 找到具体线程。
  3. 将线程 ID 转成十六进制后，用 jstack、pstack、gdb 或语言运行时自带的 profile 工具定位热点调用栈。Go 程序优先看 pprof，Java 程序优先看线程栈和 async-profiler。
  4. 如果 sy 或 si 异常高，可以用 strace -p <pid> -c 看系统调用分布，用 perf top 或 perf record 看内核热点。
  5. 如果 wa 高，继续用 iostat -x 1、iotop、pidstat -d 判断是不是磁盘 I/O 拖慢；如果网络相关，再看 sar -n DEV,TCP,ETCP 1、ss -s 和网卡中断分布。

解决方案取决于定位结果。纯计算热点要优化算法、减少无意义轮询，或者限制并发度；锁竞争要缩小临界区或降低共享状态；系统调用过多要做批量化、缓存或复用连接；I/O 等待高要优化磁盘、网络或下游服务。生产环境里最忌讳的是看到 100% 就直接重启，重启可能短暂缓解问题，但会丢掉最关键的现场信息。
